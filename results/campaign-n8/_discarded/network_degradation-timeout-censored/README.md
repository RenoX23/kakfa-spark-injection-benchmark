All 8 files here (`network_degradation_runcampaign1-8.json`, injected 2026-07-11
11:23:27Z-11:44:09Z) are the ORIGINAL network_degradation reps, discarded wholesale --
not individual outliers, the whole class was compromised.

Cause: Prometheus's `kafka-broker-jmx` scrape job had no per-job `scrape_timeout`
override, so it inherited the global default of 10s. At this class's injection severity
(500ms delay, 20% loss), 5 of the 8 reps' peak `scrape_duration_seconds` landed in a
9.8-10.01s band -- hard against that ceiling, not what an unbounded degraded-network
distribution looks like. That's a measurement ceiling silently right-censoring the
magnitude label, not genuine two-mode fault behavior (it was first misread as a bimodal
distribution before the mechanism was found). 3 of the 8 also had
`removal_confirmed: false` (the netem-removal verification race documented in
`fault_injection/network_degradation.py`'s docstring).

Fix: `infra/prometheus/values.yaml` raised `scrape_timeout` to 20s for this job
specifically (verified live via Prometheus's own `/api/v1/status/config`, not just the
committed file). `fault_injection/network_degradation.py` now permanently records a
`hit_timeout` boolean on every future rep so a censored measurement is visible in the
data instead of indistinguishable from a genuine peak.

Replacement: the class was re-run fresh at N=8 under the fixed timeout -- see
`results/campaign-n8/network_degradation/`. Note the replacement files reuse the same
`runcampaign1-8` naming and internal `run_id` values as these discarded ones (both were
generated by the same orchestrator's default run_id scheme); to make the two sets
unambiguously distinct on disk, these discarded files were renamed to a `-10s-timeout`
suffix with a matching internal `run_id` update -- see git history for this directory
if you need the exact rename commit.
